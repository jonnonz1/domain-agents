01 Why this exists the thesis

AI has collapsed the cost of building software. MVPs are cheap. But when the product takes off, the codebase hits a wall — AI-generated code optimises for now, not for scale. Worse, every new prompt lands without context: the assistant doesn't know which seams are real, which modules are hotspots, which files are fragile.

The traditional answer is "rewrite it properly". The better answer is evolutionary architecture: structure the system so each domain can evolve independently when the data says it's time, and give every AI agent editing the code the same map of those domains.

The claim. If AI is going to write most of the code, the most valuable thing a human can give it is not more prompts. It is a durable, machine-readable description of the shape of the system: where the seams are, what each seam owns, what it consumes, and when it should evolve.
What this is not. Not a code generator. Not a linter. Not "microservices in a box". It is a static analyser that produces context — agent files, rules, an MCP server — so the tools already editing your code can reason correctly about where they are.

03 How discovery works five passes, one proposal

Discovery runs five independent analysis passes, then combines the signals. Each pass can be tested alone, and each one answers a different question about the same codebase.

Pass Signal Question it answers
Structure Directories Is this codebase feature-organised, layer-organised, or mixed? Which directories behave like domain roots?
Imports TypeScript AST Which files actually depend on which? Where does the import graph cluster by connectivity? (Type-only imports weight 0.3 vs. 1.0 for value imports.)
Naming File names Do auth.controller.ts, auth.service.ts, auth.routes.ts share a prefix that groups them across layer directories?
Dependencies package.json Does stripe point to billing? Does @sendgrid/mail point to email? Does jsonwebtoken point to auth?
Interfaces Cross-domain edges Which files are imported by two or more domains? Those are boundary files — and the coupling score tells you how porous the boundary is.

Merging and hotspots

Only leaf singletons — files with no outgoing imports and exactly one consumer domain — get auto-merged during clustering. This prevents aggressive cascading that swallows unrelated domains. Singletons importing from three or more domains are flagged as hotspots and sent to unassigned; they usually indicate cross-cutting concerns a human should review.

Why five passes, not one. No single signal is reliable on its own. Directory structure lies in layer-organised codebases. Import graphs miss name-based cousins. Naming is thin in utility folders. Combining weak signals across all five passes — and being explicit about confidence — is what lets the tool handle the messy real-world case without pretending the answer is clean.

05 Editor & tooling integrations make the context load itself

Agent files are only useful if the tools editing the code can actually see them at the right moment. The hooks subcommand wires the output into the editors you already use.

Claude Code

Three layers of context

One command installs everything.

domain-agents hooks claude ./my-app
  • Per-domain rule files (.claude/rules/domain-<name>.md) with globs: frontmatter — Claude Code auto-activates the domain when you edit matching files.
  • MCP server exposing four tools: domain_lookup, domain_context, domain_files, list_domains.
  • SessionStart hook that prints a domain summary at the start of each session.
Cursor

Glob-activated rule files

Generates .cursor/rules/<domain>.mdc files. Cursor loads the relevant domain rules automatically when you open matching files.

domain-agents hooks cursor ./my-app
  • One .mdc file per domain
  • Glob patterns derived from the proposal
  • Cross-domain dependencies listed in each rule
What this feels like in practice. You say "add a new email service". Claude Code sees you editing files under the email domain, auto-loads domain-email.md, reads that email depends on users and permissions, and consults those domains too via MCP. You didn't paste anything. The context loaded itself.
06 The evolution path inline → distributed

Each agent file records where its domain currently sits on this scale, and what metric would trigger the next step. The point is not to arrive at "distributed" — it is to make the decision observable.

Stage 1
Inline
Synchronous function calls inside the monolith. Most domains live here forever.
Stage 2
Async
Fire-and-forget background work, still in-process. Triggered by latency on the critical path.
Stage 3
Queued
Durable queue (BullMQ, SQS, etc.). Triggered by retries, dead letters, or throughput spikes.
Stage 4
Separate service
Own deploy, own repo optional. Triggered by team boundaries or release-cadence divergence.
Stage 5
Distributed
Sharded, replicated, regionally placed. Triggered by scale metrics, not aesthetics.
Start consolidated, split when needed. You do not need a domain agent per micro-concern from day one. Start with broad agents covering multiple areas. Splits are cheap when the seam is already named and monitored.
07 Health — keeping the map honest drift detection · ci gates

Discovery produces a proposal. health compares the proposal to the current state of the code and tells you whether the map has drifted. The four categories:

New files
Not covered by any domain agent. Either the file needs assigning to an existing domain, or a new domain should be proposed.
High coupling
Cross-domain imports above 30%. Two domains are leaking into each other; probably one is absorbing the other, or they share a hidden dependency that wants its own domain.
Boundary violations
Cross-domain imports that bypass declared interfaces. Someone reached past the boundary file into an internal module. Fails in --ci mode.
Staleness
Domains whose code has changed since the last discovery. Signal to re-run discover and init.

CI integration

--ci exits non-zero when any of these fire. --json produces machine-readable output for custom tooling.

# .github/workflows/domain-health.yml
name: Domain Health
on:
  pull_request:
    branches: [main]
permissions:
  contents: read
jobs:
  health:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '22' }
      - run: npx domain-agents health --ci .
08 Quick start zero to integrated · 5 commands

From zero to a working, AI-tool-integrated domain map.

  1. Install globally
    Node 20 or newer. MIT licensed.
    npm install -g domain-agents
  2. Discover domains
    Static analysis only; no network. Writes .domain-agents/proposal.json — review before moving on.
    domain-agents discover ./my-app
  3. Generate agent files
    Writes agents/<domain>.md and AGENTS.md. Add --enrich to have Claude read the code and produce real descriptions instead of templates.
    domain-agents setup                  # one-time: API key for enrichment
domain-agents init ./my-app --enrich
  4. Wire up your AI tools
    Auto-activating rules plus an MCP server for Claude Code; .mdc rule files for Cursor.
    domain-agents hooks claude ./my-app
domain-agents hooks cursor ./my-app
  5. Watch for drift
    Run locally, or plug into CI on every PR.
    domain-agents health ./my-app        # human-readable report
domain-agents health --ci ./my-app   # exits non-zero on drift

10 Design principles opinions, held on purpose
Observability is the most important input

Agent files have a strong preference for gathering data. A scaling decision that is not backed by a metric is a guess. Every domain agent is asked to name the metrics that matter, and to flag the ones that are missing — because the first step in evolving a domain is usually "instrument it".

Tech debt informs feature consensus

When a feature spans multiple domains, each domain's tech debt register is consulted. A change that looks easy in isolation can be expensive once you see that two of the three domains it touches have open debt items in the exact area you're editing. Surfacing that earlier is cheap; discovering it in review is not.

Start consolidated, split when needed

A domain per micro-concern from day one is architecture cosplay. Start with broad domains covering related areas. The signal to split is real — coupling climbs, teams diverge, a subdomain accumulates its own debt — and the tool makes those signals visible. Splits happen when the data says so.

The operator resolves conflicts

Agents own their domains. But cross-domain decisions — a new interface, a debt trade-off, a scaling trigger — are surfaced to a human. The tool's job is to collapse the search space, not to decide. Hotspots, low-confidence assignments, boundary violations: all sent upward, explicitly.

Clean interfaces, simple code

The codebase itself is direct: each discovery pass is a standalone function; the orchestrator combines signals without over-coupling to any one pass; there are no unnecessary abstractions. Refactors should be possible because the seams are honest, not because a framework promised they would be.

11 FAQ the usual questions
Does it need to run my code?

No. All five discovery passes are static. The only network call is optional LLM enrichment via --enrich, which reads a handful of representative files per domain and sends them to Claude to produce contextual descriptions. You can use the tool entirely offline without it.

What about non-TypeScript codebases?

Today it is TypeScript and JavaScript (via the TypeScript compiler API, so .js, .ts, .tsx, .mjs all work). The structure, naming, and dependencies passes are language-agnostic; the imports pass is the TS-specific one. Other languages are possible but not shipped.

What if the proposal is wrong?

Edit .domain-agents/proposal.json before running init. Confidence scores and hotspot lists are there precisely so you can see where the analyser was uncertain. A bad proposal means discover needs another run — not that you're stuck with the output.

Do I need Claude Code to use this?

No. The core outputs — agents/*.md and AGENTS.md — are plain Markdown and useful on their own, including for human readers. The Claude Code and Cursor integrations are optional extras that make the files load themselves at the right moment inside those tools.

Does this replace architectural decision records?

No. Agent files are a living map; ADRs are frozen decisions. They complement each other. A well-run repo usually wants both — and the agent file is a good place to link to the relevant ADRs for each domain.

How does it handle monorepos?

Point it at a package root. Each package can have its own proposal and its own agents/ directory. A future version may discover across packages; today the unit is a single TypeScript project.
